# Role
你是一名资深的 Python 软件架构师与数据可视化专家。你擅长编写高性能、低延迟的网络通信程序，并精通使用 Python 进行复杂物理数据的处理与可视化。

# Goal
请编写一个单文件 Python 脚本（可选用 PySide6/PyQt6、Tkinter 或纯 Matplotlib 交互窗口），实现一个适用于赛车（RAC）游戏的【实时遥测数据接收、记录与多圈（Lap）对比分析工具】。
代码必须完全闭合、无 TODO 占位符、具备完整的异常处理机制，且开箱即用。

# Core Technical Requirements (核心技术要点)

为了测试你的 Python 底层功底，你必须在单文件中实现以下四个技术支柱：

## 1. 高并发异步/多线程 UDP 接收器 (Socket & Threading)
- **网络协议：** 模拟 BeamNG 或其他 RAC 游戏的遥测输出，创建一个非阻塞（Non-blocking）的 UDP Socket 监听器（默认端口 30000）。
- **线程安全：** 接收器必须运行在独立的子线程（`threading.Thread` 或 `QThread`）中，通过线程安全队列（`queue.Queue` 或信号机制）将解析后的遥测数据推送到主线程/数据缓冲池，绝对不能阻塞 UI 渲染。
- **数据帧格式：** 每帧遥测数据（JSON 或 Struct 字节流）包含以下字段：
  - `timestamp` (时间戳, float)
  - `distance` (已行驶距离/米, float)
  - `speed` (速度/km/h, float)
  - `throttle` (油门深度/0.0-1.0, float)
  - `brake` (刹车深度/0.0-1.0, float)

## 2. 核心算法：基于距离（Distance-based）的数据对齐与重采样
- **痛点解决：** 传统的“时间轴对比”在赛车遥测中是无效的（因为快的一圈时间短，慢的一圈时间长，曲线无法在横轴上对齐）。
- **算法要求：** 
  - 必须实现一个数据处理模块。当用户加载两组不同长度、不同采样率的“单圈数据记录”（Lap A 和 Lap B）时，算法必须**以“行驶距离（Distance）”为横坐标（X轴）**。
  - 使用插值算法（推荐使用 `numpy.interp` 或纯 Python 实现线性插值），将两组数据重采样（Resample）到相同的距离步长网络上，实现曲线在空间维度的精准对齐。

## 3. 实时记录与数据持久化 (Data Logging)
- 实时记录当前运行的数据。当检测到车辆重置（例如 `distance` 突变为 0 或手动触发）时，自动将当前记录的数据保存为 CSV 文件。
- CSV 格式规范：命名为 `lap_YYYYMMDD_HHMMSS.csv`，表头清晰。

## 4. 交互式对比可视化 UI (Visualization)
- **多通道曲线：** 至少绘制两个并排（或垂直堆叠）的图表：
  1. **Speed vs Distance**（速度-距离曲线）
  2. **Pedals vs Distance**（油门/刹车-距离曲线）
- **多圈对比：** 界面需要提供加载历史 CSV 的功能。加载后，图表上需同时以不同颜色（如蓝色代表 Lap A，红色代表 Lap B）绘制两条曲线，方便一眼看出“在哪里刹车早了”、“哪里出弯给油慢了”。
- **鼠标悬停十字线（Crosshair Cursor）：** 当鼠标在图表上移动时，显示一条垂直指示线，并在图例或标签中动态显示两条曲线在当前距离点上的具体数值差异。

# Mock Data Generator (内置模拟数据源)
- 为了保证代码能够独立运行测试，请在脚本中内置一个**虚拟 UDP 遥测发送器**。
- 它可以在另一个子线程中运行，每隔 20ms 发送一次模拟的车辆行驶数据（正弦规律波动的速度、逐渐累加的距离等），以便在没有连接真实游戏时，工具也能演示完整的接收、记录、绘图流程。

# Technical Constraints (技术限制)
- 仅限单文件（Single-file script）。
- 依赖库限制：标准库 + `numpy` + `matplotlib`（如果使用 GUI 框架，可选择 `PyQt6` / `PySide6` / `tkinter`）。
- 必须包含优雅的退出机制（确保子线程能安全关闭，释放 Socket 端口）。
